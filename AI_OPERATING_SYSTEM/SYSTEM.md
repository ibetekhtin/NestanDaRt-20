## 1. ГЛАВНАЯ ЦЕЛЬ

**Максимизировать КПТ = Количество Проданных Туров.**

Единственный KPI, который имеет значение.  
Все решения оцениваются через фильтр: «это увеличит КПТ?»

---

## 2. ГЛАВНЫЕ ПРИНЦИПЫ

| # | Принцип | Описание |
|---|---------|----------|
| 1 | **Простота > Красота** | Код должен быть простым, а не идеальным |
| 2 | **Минимальные изменения** | Если задачу можно решить одной строкой — не меняй десять |
| 3 | **Production First** | Всегда считать систему production. Любое изменение сохраняет работоспособность |
| 4 | **Нет новых абстракций** | Не создавать классы/сервисы/интерфейсы без объективной необходимости |
| 5 | **Доказательства вместо предположений** | Файл нельзя считать ненужным только по имени |
| 6 | **Scope Lock** | Работать только в рамках текущей задачи |
| 7 | **Откат за одну команду** | Любое изменение должно иметь `git revert` |
| 8 | **Confidence Score** | После каждого решения указывать уверенность и риски |
| 9 | **Decision Log** | Каждое решение фиксировать: почему, альтернативы, риски |
| 10 | **Think Longer** | Анализ перед изменениями важнее скорости |

## 4. КАК ПРИНИМАТЬ РЕШЕНИЯ

### Чеклист перед изменением:

1. **Уменьшает ли сложность?** (строки, файлы, зависимости, связи)
2. **Сохраняет ли поведение?** (никаких изменений UX/API)
3. **Есть ли доказательства?** (импорты, вызовы, ссылки проверены)
4. **Можно ли откатить?** (`git revert` работает)
5. **Проходит ли 5 вопросов CTO?** (см. раздел 5)

Если хотя бы один ответ «нет» — **НЕ ВЫПОЛНЯТЬ**.

---

## 5. 5 ВОПРОСОВ CTO

Каждое изменение должно пройти проверку:

1. **Станет ли архитектура проще?**
2. **Станет ли проект легче поддерживать?**
3. **Станет ли проект легче масштабировать?**
4. **Будет ли новому разработчику легче разобраться?**
5. **Будет ли система проще через 5 лет?**

Если хотя бы один ответ отрицательный — **не выполнять**.

## 7. РЕЖИМЫ РАБОТЫ

| Режим | Когда использовать | Что делает |
|-------|-------------------|------------|
| **AUDIT** | Понимание проекта | Только анализ, без изменений |
| **REFACTOR** | После аудита | Только по готовому списку |
| **BUILD** | Новая функциональность | Только построение, не удаляет |
| **SECURITY** | Безопасность | Только security-правки |
| **PERFORMANCE** | Производительность | Только perf-оптимизации |
| **RELEASE** | Перед деплоем | Build, tests, lint, version |
| **EMERGENCY** | Что-то упало | Минимальные изменения, максимальная скорость |

## 9. ГИТ ВОРКФЛОУ

| Ситуация | Действие |
|----------|----------|
| Перед этапом | `git status` — рабочий дерево должно быть чистым |
| Крупное изменение | Создать ветку `feature/xxx` |
| Перед коммитом | `git diff` — проверить изменения |
| После этапа | `git commit` с понятным сообщением |
| Откат | `git revert <commit>` — одна команда |
