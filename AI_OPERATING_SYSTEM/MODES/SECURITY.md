# SECURITY.md — РЕЖИМ БЕЗОПАСНОСТИ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Работа **только с безопасностью**.

Не трогать функциональность.  
Не трогать дизайн.  
Только security-правки.

---

## ОБЛАСТЬ ОТВЕТСТВЕННОСТИ

| Что | Где |
|-----|-----|
| Секреты | `.env`, `.env.example`, n8n credentials |
| Аутентификация | Supabase Auth, JWT, X-Kote-Secret |
| Авторизация | RLS policies, API guards |
| Валидация | Input validation, sanitization |
| Заголовки | nginx security headers, CORS |
| Сеть | UFW, firewall, открытые порты |
| Docker | Security contexts, volumes, networks |
| Зависимости | Уязвимости в requirements.txt / package.json |
| OWASP Top 10 | XSS, CSRF, SQL Injection, SSRF и т.д. |

---

## ПРАВИЛА

### 1. Безопасность > Функциональность
- [ ] Если security-фикс ломает фичу — сначала security
- [ ] Security-фикс имеет приоритет P0
- [ ] Любая уязвимость = немедленное исправление

### 2. Минимальность
- [ ] Менять только security-конфиги
- [ ] Не менять бизнес-логику
- [ ] Не менять UI/UX
- [ ] Не добавлять новую функциональность

### 3. Доказательства
- [ ] Каждый фикс ссылается на CVE / OWASP / конкретную уязвимость
- [ ] Confidence score 100% (или объяснение почему меньше)
- [ ] Проверка: что уязвимость действительно закрыта

---

## ПРОЦЕСС

### Шаг 1: Аудит безопасности
- [ ] Сканировать .env на секреты в коде
- [ ] Проверить открытые порты (UFW)
- [ ] Проверить RLS policies
- [ ] Проверить CORS
- [ ] Проверить security headers (nginx)
- [ ] Проверить валидацию входных данных
- [ ] Проверить зависимости на уязвимости (`pip audit`, `npm audit`)
- [ ] Проверить SQL-инъекции (все запросы через параметры)
- [ ] Проверить XSS (все выводы экранированы)
- [ ] Проверить CSRF (если есть формы)

### Шаг 2: Классификация
| Severity | Описание | Действие |
|----------|----------|----------|
| **Critical** | RCE, SQLi, утечка данных | Немедленно |
| **High** | XSS, CSRF, auth bypass | В течение дня |
| **Medium** | Information disclosure | В течение недели |
| **Low** | Security headers, best practices | В течение месяца |

### Шаг 3: Исправление
Для каждой уязвимости:
1. Описать проблему
2. Описать решение
3. Применить минимальное изменение
4. Проверить: уязвимость закрыта
5. Проверить: функциональность не сломана
6. Commit: `git commit -m "security: описание"`

### Шаг 4: Отчёт
Заполнить `REPORTS/SECURITY_<дата>.md`:
- Найденные уязвимости
- Исправления
- Confidence score
- Риски

## ЧТО ЗАПРЕЩЕНО

| Запрещено | Почему |
|-----------|--------|
| Менять бизнес-логику | Это не security |
| Менять UI/UX | Это не security |
| Добавлять новую функциональность | Это не security |
| Упрощать security "для удобства" | Риск уязвимости |
| Отключать RLS "для теста" | Критическая уязвимость |
| Публиковать секреты | Даже в .env.example |

## ПРИОРИТЕТЫ

| Приоритет | Тип |
|-----------|-----|
| P0 | Утечка секретов (.env в git, хардкод) |
| P0 | SQL Injection |
| P0 | XSS (отражённый/хранимый) |
| P0 | Auth bypass |
| P1 | CSRF |
| P1 | Открытые порты (UFW) |
| P1 | Слабые RLS policies |
| P2 | Security headers (CSP, HSTS) |
| P2 | CORS overly permissive |
| P2 | Устаревшие зависимости |

---

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед каждым изменением:
1. **Это security-фикс?** (не рефакторинг)
2. **Это минимальное изменение?** (не "улучшение")
3. **Уязвимость доказана?** (не предположение)
4. **Фикс не ломает функциональность?** (всё работает)
5. **Можно откатить?** (`git revert` работает)

Если хотя бы один ответ «нет» — **НЕ ВЫПОЛНЯТЬ**.
