# REFACTOR.md — РЕЖИМ РЕФАКТОРИНГА
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Выполнение рефакторинга **только по готовому аудиту**.

Не принимать собственных решений о том, что менять.  
Только выполнять проверенный список.

---

## ПРАВИЛА

### 1. Входные данные
- [ ] Обязательно иметь `REPORTS/AUDIT_<дата>.md`
- [ ] В аудите указаны конкретные файлы/строки для изменения
- [ ] Для каждого изменения указан Confidence score
- [ ] Для каждого изменения указаны риски

### 2. Безопасность
- [ ] Каждое изменение проверяет импорты
- [ ] Каждое изменение проверяет вызовы
- [ ] Каждое изменение проверяет зависимости
- [ ] После изменения запускает backend
- [ ] После изменение запускает frontend
- [ ] После изменение проверяет основные страницы

### 3. Минимальность
- [ ] Изменять только указанные в аудите файлы
- [ ] Не добавлять новую функциональность
- [ ] Не менять бизнес-логику
- [ ] Не менять пользовательское поведение
- [ ] Не переписывать код ради красоты

### 4. Откат
- [ ] Перед изменением: `git status` (должно быть чисто)
- [ ] Перед изменением: `git branch` (создать ветку если крупное)
- [ ] После каждого изменения: `git diff` (проверить)
- [ ] После этапа: `git commit` (зафиксировать)
- [ ] Откат: `git revert <commit>` (одна команда)

---

## ПРОЦЕСС

### Шаг 1: Подготовка
```bash
git status                    # Должно быть чисто
git checkout -b refactor/xxx  # Если крупное изменение
```

### Шаг 2: Выполнение (по одному)
Для каждого пункта из аудита:
1. Прочитать файл
2. Проверить контекст (импорты, вызовы)
3. Внести минимальное изменение
4. Проверить: `git diff`
5. Запустить backend/frontend
6. Проверить работоспособность
7. Commit: `git commit -m "refactor: описание"`

### Шаг 3: Проверка
- [ ] Все изменения из аудита выполнены
- [ ] Никаких лишних изменений
- [ ] Backend запускается
- [ ] Frontend собирается
- [ ] Основные страницы открываются
- [ ] Нет ошибок в консоли

### Шаг 4: Отчёт
Заполнить `REPORTS/REFACTOR_<дата>.md`:
- Что изменено
- Почему безопасно
- Проверенные зависимости
- Confidence score
- Риски

## ЧТО ЗАПРЕЩЕНО

| Запрещено | Почему |
|-----------|--------|
| Принимать собственные решения | Только по аудиту |
| Удалять код без доказательств | Аудит — доказательство |
| Менять архитектуру | Только оптимизация |
| Добавлять новое | Только удаление/объединение |
| Переписывать функции | Только рефакторинг |
| Менять имена без необходимости | Риск сломать вызовы |

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед каждым изменением:
1. **Это уменьшает сложность?** (строки, файлы, зависимости)
2. **Это сохраняет поведение?** (UX/API не меняются)
3. **Есть доказательства?** (аудит подтверждает)
4. **Можно откатить?** (`git revert` работает)
5. **Проходит 5 вопросов CTO?** (см. SYSTEM.md)

Если хотя бы один ответ «нет» — **НЕ ВЫПОЛНЯТЬ**.
