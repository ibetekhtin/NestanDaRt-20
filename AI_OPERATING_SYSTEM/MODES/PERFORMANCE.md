# PERFORMANCE.md — РЕЖИМ ПРОИЗВОДИТЕЛЬНОСТИ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Работа **только с производительностью**.

Не трогать архитектуру.  
Не трогать функциональность.  
Только perf-оптимизации.

---

## ОБЛАСТЬ ОТВЕТСТВЕННОСТИ

| Что | Где | Метрики |
|-----|-----|---------|
| Загрузка страниц | HTML/CSS/JS | LCP, FID, CLS (Lighthouse) |
| Размер бандла | JS/CSS | Bundle size, gzip/brotli |
| Время ответа API | FastAPI | TTFB, latency |
| Рендеринг | React, CSS | FPS, repaints |
| Изображения | img, SVG | WebP, lazy load, compression |
| Кэширование | nginx, headers | Cache-Control, ETag |
| Запросы | API, DB | Количество, размер |

---

## ПРАВИЛА

### 1. Производительность > Красота
- [ ] Если perf-оптимизация ухудшает читаемость — допустимо
- [ ] Perf-фикс имеет приоритет P1
- [ ] Любое увеличение размера бандла = оправдание

### 2. Измеримость
- [ ] Каждая оптимизация должна иметь метрику "до"
- [ ] Каждая оптимизация должна иметь метрику "после"
- [ ] Confidence score на основе измерений, а не предположений

### 3. Минимальность
- [ ] Оптимизировать только измеренные проблемы
- [ ] Не оптимизировать "на будущее"
- [ ] Не добавлять сложность "для скорости"
- [ ] Использовать простые решения

---

## ПРОЦЕСС

### Шаг 1: Измерение
- [ ] Lighthouse CI (или локально)
- [ ] Bundle analyzer (webpack-bundle-analyzer, source-map-explorer)
- [ ] Network tab (DevTools) — количество запросов, размер
- [ ] API latency (curl, wrk)
- [ ] Database queries (EXPLAIN ANALYZE)

### Шаг 2: Приоритизация
| Метрика | Цель | Действие если не проходит |
|---------|------|---------------------------|
| LCP | < 2.5s | Оптимизировать изображения, кэш |
| FID | < 100ms | Уменьшить JS, code splitting |
| CLS | < 0.1 | Фиксировать размеры изображений |
| Bundle JS | < 200 KB | Удалить неиспользуемый код |
| Bundle CSS | < 50 KB | Удалить неиспользуемые стили |
| TTFB | < 500ms | Кэш, CDN, DB индексы |

### Шаг 3: Оптимизация
Для каждой проблемы:
1. Описать проблему (метрика "до")
2. Предложить решение
3. Применить минимальное изменение
4. Измерить результат (метрика "после")
5. Проверить: функциональность не сломана
6. Commit: `git commit -m "perf: описание"`

### Шаг 4: Отчёт
Заполнить `REPORTS/PERFORMANCE_<дата>.md`:
- Метрики "до"
- Метрики "после"
- Что оптимизировано
- Confidence score
- Риски

## ЧТО ЗАПРЕЩЕНО

| Запрещено | Почему |
|-----------|--------|
| Менять архитектуру | Это не perf-режим |
| Менять бизнес-логику | Это не perf-режим |
| Добавлять новую функциональность | Это не perf-режим |
| Удалять код "для скорости" без измерений | Предположение |
| Создавать сложные кэши "на будущее" | Усложнение |
| Оптимизировать то, что не измерено | Неизвестный эффект |

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед каждым изменением:
1. **Это perf-оптимизация?** (не рефакторинг)
2. **Проблема измерена?** (есть метрика "до")
3. **Это минимальное изменение?** (не "улучшение")
4. **Фикс не ломает функциональность?** (всё работает)
5. **Можно откатить?** (`git revert` работает)

Если хотя бы один ответ «нет» — **НЕ ВЫПОЛНЯТЬ**.
