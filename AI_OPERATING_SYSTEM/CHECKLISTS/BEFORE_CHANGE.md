# BEFORE_CHANGE.md — ПЕРЕД ЛЮБЫМ ИЗМЕНЕНИЕМ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Обязательная проверка перед любым изменением кода.

Не пропускать.  
Не оптимизировать.  
Выполнять полностью.

---

## ПРОВЕРКИ

### 1. Git
- [ ] `git status` — рабочее дерево чистое
- [ ] `git log --oneline -3` — последние коммиты понятны
- [ ] Если крупное изменение — создана ветка

### 2. Понимание
- [ ] Прочитал задачу полностью
- [ ] Прочитал связанный код
- [ ] Понимаю, что должно измениться
- [ ] Понимаю, что НЕ должно измениться

### 3. Зависимости
- [ ] Нашёл все импорты, которые затронуты
- [ ] Нашёл все вызовы функций, которые затронуты
- [ ] Нашёл все ссылки в HTML/CSS/JS
- [ ] Нашёл все упоминания в конфигах (nginx, docker)
- [ ] Нашёл все упоминания в документации

### 4. Безопасность
- [ ] Изменение не ломает API
- [ ] Изменение не ломает backend
- [ ] Изменение не ломает frontend
- [ ] Изменение не ломает Docker
- [ ] Изменение не ломает деплой
- [ ] Изменение не ломает nginx
- [ ] Изменение не ломает Supabase
- [ ] Изменение не ломает n8n

### 5. Откат
- [ ] Изменение можно откатить (`git revert`)
- [ ] Есть план отката (если крупное)
- [ ] Есть бэкап (если крупное)

### 6. 5 Вопросов CTO
- [ ] Архитектура станет проще? (не усложнится)
- [ ] Проект станет легче поддерживать? (не сложнее)
- [ ] Проект станет легче масштабировать? (не сложнее)
- [ ] Новому разработчику будет проще? (не сложнее)
- [ ] Система станет проще через 5 лет? (не сложнее)

### 7. Scope Lock
- [ ] Изменение только в рамках задачи
- [ ] Не трогаю соседние модули
- [ ] Не исправляю "заодно" другие баги
- [ ] Не улучшаю "пока я тут"

---

## ЕСЛИ ХОТЯ БЫ ОДИН ОТВЕТ «НЕТ»

**НЕ ВЫПОЛНЯТЬ ИЗМЕНЕНИЕ.**

Вернуться к анализу.  
Устранить проблему.  
Повторить проверку.

---

## КОНТРОЛЬНЫЙ ВОПРОС

Перед изменением спроси себя:

> «Если это изменение сломает что-то — я пойму, что именно, и смогу откатить за 1 минуту?»

Если ответ «нет» — **НЕ НАЧИНАТЬ**.
