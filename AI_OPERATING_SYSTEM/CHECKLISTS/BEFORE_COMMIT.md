# BEFORE_COMMIT.md — ПЕРЕД КОММИТОМ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Проверка перед каждым коммитом.

Не коммитить "на всякий случай".  
Только проверенное.

---

## ПРОВЕРКИ

### 1. Что коммитирую
- [ ] `git diff --staged` — посмотрел изменения
- [ ] Изменения соответствуют сообщению коммита
- [ ] Нет лишних файлов (нечаянно добавленных)
- [ ] Нет секретов (.env, ключи, пароли)

### 2. Качество
- [ ] Код проходит линтер (flake8 / eslint)
- [ ] Нет console.log (только console.error)
- [ ] Нет закомментированного кода
- [ ] Нет TODO (если не связано с задачей)

### 3. Зависимости
- [ ] Если добавил зависимость — обосновал
- [ ] Если обновил зависимость — проверил
- [ ] requirements.txt / package.json обновлены

### 4. Документация
- [ ] Если менял API — обновил docs/API.md
- [ ] Если менял архитектуру — обновил PROJECT_MAP.md
- [ ] Если менял env — обновил docs/ENV.md

### 5. Сообщение
- [ ] Сообщение понятное (не "update", "fix")
- [ ] Type правильный (feat/fix/refactor/docs)
- [ ] Описание соответствует изменениям

---

## ФОРМАТ КОММИТА

```bash
git commit -m "<type>: <описание>"
```

### Types
| Type | Пример |
|------|--------|
| `feat` | `feat: add Pattaya landing page` |
| `fix` | `fix: auth token expiration` |
| `refactor` | `refactor: remove duplicate PWA files` |
| `perf` | `perf: optimize image loading` |
| `docs` | `docs: update API documentation` |
| `security` | `security: fix SQL injection in tours` |
| `chore` | `chore: update dependencies` |

---

## ЕСЛИ ХОТЯ БЫ ОДИН ОТВЕТ «НЕТ»

**НЕ КОММИТИТЬ.**

Исправить проблему.  
Повторить проверку.

---

## КОНТРОЛЬНЫЙ ВОПРОС

> «Если я откачу этот коммит — система будет работать?»

Если ответ «нет» — **НЕ КОММИТИТЬ**.
