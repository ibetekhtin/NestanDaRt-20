# GIT.md — ПРАВИЛА РАБОТЫ С GIT
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Git — инструмент для отката и истории.

Не для "красивого коммита".  
Не для "идеальной истории".  
Для безопасности и отката.

---

## ПРИНЦИПЫ

### 1. Откат за одну команду
- [ ] Любое изменение должно иметь `git revert`
- [ ] Не делать коммиты, которые нельзя откатить
- [ ] Не делать "WIP" коммиты в main

### 2. Чистое рабочее дерево
- [ ] Перед каждым этапом — `git status` (должно быть чисто)
- [ ] Не начинать новое изменение, пока старое не закоммичено
- [ ] Не смешивать разные задачи в одном коммите

### 3. Минимальные коммиты
- [ ] Один коммит = одна логическая единица
- [ ] Не коммитить "и ещё кое-что"
- [ ] Не коммитить "пока я тут"

## ВЕТКИ

### Когда создавать ветку
| Ситуация | Ветка |
|----------|-------|
| Крупное изменение (3+ файлов) | `feature/xxx` |
| Рефакторинг | `refactor/xxx` |
| Исправление бага | `fix/xxx` |
| Release | `release/vX.Y.Z` |

### Когда НЕ создавать ветку
- Небольшое изменение (1-2 файла)
- Исправление опечатки
- Обновление документации

### Имена веток
```
feature/add-pattaya-landing
refactor/remove-duplicates
fix/auth-token-expired
release/v1.2.0
```

### Запрещено
- Создавать ветки "на всякий случай"
- Держать ветки дольше недели
- Создавать ветки с именами `test`, `tmp`, `temp`

## КОММИТЫ

### Формат
```
<type>: <описание>

[тело опционально]

[footer опционально]
```

### Types
| Type | Когда использовать |
|------|-------------------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `refactor` | Рефакторинг (не меняет поведение) |
| `perf` | Оптимизация производительности |
| `docs` | Только документация |
| `security` | Security-фикс |
| `chore` | Обновление зависимостей, конфигов |
| `revert` | Откат коммита |

### Примеры
```bash
# ✅ Хорошо
git commit -m "feat: add Pattaya landing page"
git commit -m "fix: auth token expiration"
git commit -m "refactor: remove duplicate PWA files"
git commit -m "docs: update API documentation"

# ❌ Плохо
git commit -m "update"
git commit -m "fix"
git commit -m "WIP"
git commit -m "changes"
```

### Запрещено
- Коммитить без сообщения
- Коммитить с сообщением "update", "fix", "changes"
- Делать коммиты > 500 строк (разбить)
- Коммитить секреты (.env)

## РАБОЧИЙ ПРОЦЕСС

### Перед началом работы
```bash
git status              # Должно быть чисто
git pull origin main    # Актуальная main
```

### Во время работы
```bash
git add <файлы>         # Добавить конкретные файлы
git status              # Проверить что добавили
git diff --staged       # Проверить изменения
git commit -m "..."     # Закоммитить
```

### После этапа
```bash
git log --oneline -5    # Проверить историю
git push origin main    # Отправить (если нужно)
```

### Откат
```bash
# Откат последнего коммита (сохраняет изменения)
git reset --soft HEAD~1

# Откат последнего коммита (удаляет изменения)
git reset --hard HEAD~1

# Откат конкретного коммита
git revert <commit_hash>

# Откат файла к последнему коммиту
git checkout HEAD -- <файл>
```

## .gitignore

### Обязательно игнорировать:
```
# Секреты
.env
.env.local
*.key
*.pem

# Зависимости
node_modules/
venv/
__pycache__/
*.pyc

# Сборки
dist/
build/
*.egg-info/

# IDE
.vscode/
.idea/
*.swp

# Система
.DS_Store
Thumbs.db

# Логи
*.log
logs/

# Кэш
.cache/
tmp/
temp/
```

### Запрещено коммитить:
- `.env` (даже `.env.example` с реальными значениями)
- `node_modules/`, `venv/`
- Секреты, ключи, пароли
- Логи с чувствительными данными

## TAG (Версионирование)

### Когда создавать тег
- Перед релизом в production
- После стабильного коммита

### Формат
```
v1.2.3
```

### Команды
```bash
# Создать тег
git tag -a v1.2.3 -m "Release v1.2.3"

# Отправить тег
git push origin v1.2.3

# Список тегов
git tag -l

# Проверить тег
git show v1.2.3
```

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед коммитом:
1. **Это одна логическая единица?** (не смесь задач)
2. **Сообщение понятное?** (не "update")
3. **Не коммичу ли секреты?** (проверил .gitignore)
4. **Это можно откатить?** (`git revert` работает)
5. **Рабочее дерево чистое?** (нет лишних файлов)

Если хотя бы один ответ «нет» — **НЕ КОММИТИТЬ**.

## ПРИМЕРЫ

### ✅ Хорошо
```bash
# 1. Проверил статус
git status

# 2. Добавил конкретные файлы
git add nestandart-phuket/index.html nestandart-phuket/css/style.css

# 3. Проверил изменения
git diff --staged

# 4. Закоммитил с понятным сообщением
git commit -m "feat: add Pattaya section to landing page"
```

### ❌ Плохо
```bash
# Добавил всё подряд
git add .

# Закоммитил без сообщения
git commit -m "update"

# Или с неясным сообщением
git commit -m "fix stuff"
```
