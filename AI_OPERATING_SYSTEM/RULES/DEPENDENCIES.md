## НАЗНАЧЕНИЕ

Управление зависимостями без усложнения.

Не добавлять пакеты "на будущее".  
Не обновлять "просто так".  
Только когда есть причина.

---

## ПРИНЦИПЫ

### 1. Минимализм
- [ ] Каждая зависимость должна использоваться
- [ ] Каждая зависимость должна обосновываться
- [ ] Если можно без зависимости — не добавлять

### 2. Стабильность
- [ ] Фиксировать версии (не `*`, не `latest`)
- [ ] Не обновлять без причины
- [ ] Тестировать после обновления

### 3. Безопасность
- [ ] Проверять уязвимости (`pip audit`, `npm audit`)
- [ ] Обновлять только security-фиксы
- [ ] Не использовать deprecated пакеты

## PYTHON (requirements.txt)

### Добавление новой зависимости
1. **Есть ли стандартная библиотека?** (например, `json`, `datetime`, `pathlib`)
2. **Есть ли встроенное FastAPI?** (например, `pydantic`, `sqlalchemy`)
3. **Если нет — обосновать:** что делает, почему нельзя без неё, альтернативы

### Обновление зависимости
- [ ] Только security fix (CVE)
- [ ] Только если deprecated
- [ ] Только если сломано

### Удаление зависимости
- [ ] Проверить: нигде не импортируется
- [ ] Проверить: нигде не упоминается в docs
- [ ] Confidence 100%

### Запрещено
- Добавлять "популярные" пакеты без причины
- Добавлять пакеты "на будущее"
- Обновлять "до последней версии"
- Использовать `*` или `>=` без верхней границы

## JAVASCRIPT (package.json)

### Добавление новой зависимости
1. **Есть ли нативный API?** (например, `fetch`, `Promise`, `URL`)
2. **Есть ли в React?** (например, `useState`, `useEffect`)
3. **Если нет — обосновать:** что делает, почему нельзя без неё, размер (gzip)

### Обновление зависимости
- [ ] Только security fix
- [ ] Только если deprecated
- [ ] Только если сломано

### Удаление зависимости
- [ ] Проверить: нигде не импортируется
- [ ] Проверить: нигде не упоминается в docs
- [ ] Confidence 100%

### Запрещено
- Добавлять lodash, moment, axios (есть нативные аналоги)
- Добавлять UI-киты (если нужен 1 компонент)
- Добавлять пакеты "на будущее"
- Использовать `*` или `^` без контроля

## DOCKER

### Образы
- [ ] Фиксировать версии (не `latest`)
- [ ] Использовать alpine/slim где возможно
- [ ] Не добавлять лишние слои

### Пример
```yaml
# ✅ Хорошо
services:
  backend:
    image: python:3.11-slim
    # конкретная версия, минимальный размер

# ❌ Плохо
services:
  backend:
    image: python:latest
    # нестабильно, может сломаться
```

### Запрещено
- Использовать `latest` тег
- Создавать кастомные образы "для удобства"
- Добавлять инструменты в production образ (curl, vim и т.д.)

## VERSIONING

### Семантическое версионирование
- `MAJOR.MINOR.PATCH` (например, 1.2.3)
- MAJOR — ломающие изменения
- MINOR — новая функциональность
- PATCH — исправления

### Где фиксировать версии
- `requirements.txt` — все пакеты
- `package.json` — все пакеты
- `docker-compose.yml` — образы
- `.env.example` — версии API (если есть)

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед добавлением зависимости:
1. **Это используется?** (не "может понадобиться")
2. **Это нельзя сделать без неё?** (стандартная библиотека?)
3. **Это минимальное решение?** (не "удобное")
4. **Это безопасно?** (нет уязвимостей)
5. **Это можно откатить?** (`git revert` работает)

Если хотя бы один ответ «нет» — **НЕ ДОБАВЛЯТЬ**.
