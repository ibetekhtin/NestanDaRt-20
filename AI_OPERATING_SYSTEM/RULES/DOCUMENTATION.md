# DOCUMENTATION.md — ПРАВИЛА ДОКУМЕНТАЦИИ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Документация должна быть актуальной.

Не писать документацию "для галочки".  
Не обновлять "потом".  
Только когда код изменился.

---

## ПРИНЦИПЫ

### 1. Код — источник истины
- [ ] Документация описывает то, что есть в коде
- [ ] Документация не описывает то, что было удалено
- [ ] Документация не описывает "планы на будущее"

### 2. Минимальность
- [ ] Документировать только то, что не очевидно из кода
- [ ] Не дублировать код в документации
- [ ] Не писать "как использовать" для очевидных вещей

### 3. Актуальность
- [ ] При каждом изменении API — обновить `docs/API.md`
- [ ] При каждом изменении архитектуры — обновить `PROJECT_MAP.md`
- [ ] При каждом изменении Docker — обновить `README.md`
- [ ] При каждом изменении env — обновить `docs/ENV.md`

---

## ЧТО ДОКУМЕНТИРОВАТЬ

### Обязательно документировать:
| Что | Где | Когда |
|-----|-----|-------|
| API endpoints | `docs/API.md` | При добавлении/изменении |
| Архитектура | `PROJECT_MAP.md` | При изменении структуры |
| Переменные окружения | `docs/ENV.md` | При добавлении новой |
| Docker compose | `README.md` | При изменении сервисов |
| Миграции БД | `docs/MIGRATIONS.md` (если есть) | При каждой миграции |
| n8n workflows | `docs/N8N.md` | При изменении workflow |

### Не документировать:
- Очевидный код (имена говорят сами за себя)
- Временные фичи (документировать когда станут постоянными)
- "Планы на будущее" (не в документации)
- Очевидные команды (`git status`, `npm install`)

## ОБНОВЛЕНИЕ ДОКУМЕНТАЦИИ

### При изменении кода:
1. **Изменил API?** → Обновить `docs/API.md`
2. **Изменил архитектуру?** → Обновить `PROJECT_MAP.md`
3. **Изменил Docker?** → Обновить `README.md`
4. **Изменил env?** → Обновить `docs/ENV.md`
5. **Изменил безопасность?** → Обновить `docs/SECURITY.md` (если есть)

### Процесс:
1. Внести изменение в код
2. Сразу обновить документацию
3. Commit вместе: `git commit -m "feat: описание + docs"`
4. Не оставлять "документацию на потом"

## ЧТО ЗАПРЕЩЕНО

| Запрещено | Почему |
|-----------|--------|
| Документировать "планы на будущее" | Неактуально |
| Дублировать код в документации | Два источника истины |
| Оставлять устаревшие разделы | Вводит в заблуждение |
| Писать "TODO" в документации | Код — для TODO |
| Создавать документацию "для галочки" | Не читают |
| Обновлять документацию "потом" | Забудется |

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед коммитом:
1. **Документация обновлена?** (если менял API/архитектуру/env)
2. **Документация актуальна?** (не описывает удалённое)
3. **Документация минимальна?** (не дублирует код)
4. **Документация читаема?** (новый разработчик поймёт)
5. **Документация в одном месте?** (не разбросана по 5 файлам)

Если хотя бы один ответ «нет» — **НЕ КОММИТИТЬ**.
