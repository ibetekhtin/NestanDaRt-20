# ARCHITECTURE.md — ПРАВИЛА АРХИТЕКТУРЫ
> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Сохранение простоты архитектуры.

Не усложнять без необходимости.  
Не создавать абстракции "на будущее".  
Не разносить код по папкам "для порядка".

---

## ПРИНЦИПЫ

### 1. Простота > Идеальность
- [ ] Архитектура должна быть понятна новому разработчику за 1 час
- [ ] Не более 3 уровней вложенности папок
- [ ] Не более 5 файлов в папке (если не логическая группа)
- [ ] Каждый файл должен иметь одну ответственность

### 2. Минимальные изменения
- [ ] Если можно решить в одном файле — не создавать новый
- [ ] Если можно решить в одной функции — не создавать класс
- [ ] Если можно решить без абстракции — не создавать интерфейс
- [ ] Если можно решить без сервиса — не создавать сервис

### 3. Существующие паттерны
- [ ] Использовать уже существующие паттерны в проекте
- [ ] Не изобретать велосипед
- [ ] Не создавать "универсальные" решения для одной задачи

## СТРУКТУРА ПРОЕКТА (канон)

```
NestanDaRt-20/
├── app/backend/          ← FastAPI (роутеры по domains)
├── providers/            ← AI fallback chain
├── nestandart-phuket/    ← Сайт (HTML/CSS/JS)
├── baza/                   ← React/Vite (компоненты по фичам)
├── deploy/               ← Скрипты деплоя
├── docs/                 ← Документация
├── supabase/             ← Миграции
├── scripts/              ← Вспомогательные скрипты
└── shared/               ← Общий код (только если используется 3+ места)
```

### Запрещено:
- Создавать папки `utils/`, `helpers/`, `common/` без доказательств использования
- Создавать папки `services/`, `repositories/`, `controllers/` без необходимости
- Разносить один компонент на 5 файлов (jsx, css, test, story, types)

## ЗАВИСИМОСТИ МЕЖДУ МОДУЛЯМИ

| Модуль | Может зависеть от | Не может зависеть от |
|--------|-------------------|----------------------|
| `shared/` | Ничего | Всё (только общий код) |

### Правило:
- **Backend не знает о frontend**
- **Frontend не знает о backend** (только через API)
- **PWA не знает о сайте** (только через Supabase)
- **HQ не знает о клиентах** (только через Supabase)

### Запрещено:
- Менять существующие endpoints без обсуждения
- Менять структуру response без обсуждения

- Документировать каждый endpoint в `docs/API.md`

### Запрещено:
- Создавать компоненты "на будущее"
- Создавать компоненты с одним использованием
- Разносить один компонент на 5 файлов
- Использовать пропсы drilling (использовать Context)

### Обязательно:
- Компонент = один файл (если не логическая группа)
- Компонент < 200 строк (иначе разбить)
- Использовать существующие паттерны (View, Context)
- Не создавать абстракции "для переиспользования"

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед каждым архитектурным изменением:
1. **Это упрощает архитектуру?** (не усложняет)
2. **Это уменьшает количество файлов?** (не увеличивает)
3. **Это уменьшает количество зависимостей?** (не увеличивает)
4. **Это использует существующие паттерны?** (не изобретает)
5. **Новому разработчику будет проще?** (не сложнее)

Если хотя бы один ответ «нет» — **НЕ ВЫПОЛНЯТЬ**.

```python
# Один файл, одна функция, простота

    return supabase.table('tours').select('*').eq('market_id', market_id).execute().data
```
