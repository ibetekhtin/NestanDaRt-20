> Версия: 1.0

---

## НАЗНАЧЕНИЕ

Фиксация каждого важного решения.

Почему принято.  
Какие альтернативы.  
Почему не они.  
Какие риски.  
Какие компромиссы.

---

## ЗАЧЕМ ЭТО НУЖНО

- Через полгода забывается, почему было принято решение
- Новый разработчик не понимает логику
- При повторной проблеме — повторяются старые ошибки
- При аудите — видна история изменений

---

## ФОРМАТ

Каждое решение записывается в таком формате:

```markdown
## [DEC-001] Название решения
**Дата:** YYYY-MM-DD  
**Автор:** Имя / AI-агент  
**Статус:** Принято / Отклонено / Отложено

### Контекст
Что произошло, какая проблема, какой вопрос.

### Решение
Что конкретно решено.

### Альтернативы
1. **Альтернатива 1** — почему не выбрана
2. **Альтернатива 2** — почему не выбрана

### Риски
- Риск 1 — как mitigations
- Риск 2 — как mitigations

### Компромиссы
Что приняли "не идеально", но "достаточно хорошо".

### Confidence
98% — почему уверен, 2% — что может быть неверным.

### Проверка
Как проверить, что решение работает.

### Ссылки
- CLAUDE.md раздел X
- docs/API.md
- GitHub issue #123
```

## КОГДА ЗАПИСЫВАТЬ

### Обязательно:
- Изменение архитектуры
- Добавление нового API endpoint
- Изменение схемы БД
- Удаление файлов/функций
- Изменение зависимостей
- Security-фикс
- Изменение конфигурации Docker/nginx

### Опционально:
- Рефакторинг (если > 3 файлов)
- Оптимизация производительности
- Изменение дизайна

### Не нужно:
- Исправление опечаток
- Небольшие правки (< 5 строк)
- Обновление документации

## ПРИМЕРЫ

### Пример 1: Удаление дубликата

```markdown
**Дата:** 2026-06-29  
**Автор:** AI-агент  
**Статус:** Принято

### Контекст
Найдено 3 версии PWA: app.html, app-hyper.html, public/index.html.
Все три содержат одинаковый код с минимальными отличиями.

### Решение
Удалить app-hyper.html и public/index.html.
Оставить app.html как каноническую версию (v11.0).

### Альтернативы
1. **Оставить все три** — дублирование, сложность поддержки
2. **Объединить в один** — требует времени, не уменьшает сложность
3. **Удалить app.html** — это основная версия, используется в production

### Риски
- Кто-то может использовать app-hyper.html — проверены все импорты, 0 зависимостей
- Могут быть ссылки в документации — проверены docs/, 0 упоминаний

### Компромиссы
Если кому-то нужен hyper-режим — можно добавить флаг в app.html.

### Confidence
100% — все импорты проверены, 0 зависимостей.

### Проверка
Запустить PWA, убедиться что работает.

### Ссылки
- CODE_AUDIT.md раздел 1.1
- PROJECT_MAP.md раздел 6.1
```

### Пример 2: Добавление нового API

```markdown
## [DEC-002] Добавление /api/v1/ping
**Дата:** 2026-06-29  
**Автор:** AI-агент  
**Статус:** Принято

### Контекст
Нужен простой health check для мониторинга.

### Решение
Добавить GET /api/v1/ping → возвращает {"status": "ok", "timestamp": "..."}

### Альтернативы
1. **Использовать /health** — уже есть, но возвращает больше данных
2. **Использовать /api/v1/health** — то же самое
3. **Создать /ping** — минимальный, только статус

### Риски
- Новый endpoint — нужно документировать
- Может быть滥用 для DDoS — добавить rate limit (опционально)

### Компромиссы
Не добавляем rate limit сейчас — если будет проблема, добавим потом.

### Confidence
95% — простой endpoint, минимальные риски.

### Проверка
curl http://localhost:8000/api/v1/ping
```

## ПРАВИЛА

### Запрещено:
- Принимать решение без записи (забудется)
- Записывать "потом" (забудется контекст)
- Писать "очевидно" (не очевидно через полгода)
- Удалять старые решения (история важна)

### Обязательно:
- Записывать сразу после принятия
- Указывать Confidence score
- Указывать альтернативы
- Указывать риски
- Ссылаться на код/документы

---

## ИНТЕГРАЦИЯ С РЕЖИМАМИ

| Режим | Когда записывать |
|-------|-----------------|
| AUDIT | Каждая найденная проблема с Confidence |
| REFACTOR | Каждое удаление/объединение |
| BUILD | Каждый новый endpoint/компонент |
| SECURITY | Каждый security-фикс |
| PERFORMANCE | Каждая оптимизация с метриками |
| RELEASE | Каждый релиз с планом отката |
| EMERGENCY | Каждый фикс с причиной |

---

## КОНТРОЛЬНЫЕ ВОПРОСЫ

Перед принятием решения:
1. **Это записано?** (не в голове)
2. **Это можно откатить?** (если ошибся)
3. **Это документировано?** (в DECISIONS.md)
4. **Это объяснено?** (новый разработчик поймёт)
5. **Это ссылается на код?** (можно найти)

Если хотя бы один ответ «нет» — **ЗАПИСАТЬ ПЕРЕД ВЫПОЛНЕНИЕМ**.
